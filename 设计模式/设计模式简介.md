# 设计模式简介

## 一、设计模式是什么

设计模式，即Design Patterns，是指在软件开发过程中，被反复使用的代码设计经验，用于提高代码的可重用性和可维护性

## 二、设计模式的类型

### 1. 创建型模式（Creational）

👉 关注“对象如何创建”

- 单例模式（Singleton）
- 工厂方法模式（Factory Method）
- 抽象工厂模式（Abstract Factory）
- 建造者模式（Builder）
- 原型模式（Prototype）

### 2. 结构型模式（Structural）

👉 关注“类和对象如何组合”

- 适配器模式（Adapter）
- 装饰器模式（Decorator）
- 代理模式（Proxy）
- 外观模式（Facade）
- 桥接模式（Bridge）
- 组合模式（Composite）
- 享元模式（Flyweight）

### 3. 行为型模式（Behavioral）

👉 关注“对象之间如何通信”

- 观察者模式（Observer）
- 策略模式（Strategy）
- 模板方法模式（Template Method）
- 责任链模式（Chain of Responsibility）
- 命令模式（Command）
- 状态模式（State）
- 迭代器模式（Iterator）
- 中介者模式（Mediator）
- 备忘录模式（Memento）
- 解释器模式（Interpreter）
- 访问者模式（Visitor）

## 三、设计模式的核心原则—SOLID

### 1、单一职责原则 (Single Responsibility Principle, SRP)

一个类的功能应该尽可能单一，如果有多个功能，则要定义多个类

**场景**：一个 `Order` 类最初包含订单的商品明细、计算总价、保存到数据库、发送邮件通知等多个功能。

```java
// 违反 SRP：一个类承担了太多责任
public class Order {
    public void calculateTotal() { ... }
    public void saveToDatabase() { ... }
    public void sendEmail() { ... }
}
```

**问题**：当邮件服务器变更或数据库表结构调整时，都要修改 `Order` 类，而且修改邮件逻辑可能意外影响订单计算。

**重构**：拆分为三个类，各司其职：

- `OrderCalculator` 负责金额计算
- `OrderRepository` 负责持久化
- `EmailService` 负责发送邮件

这样每个类只有一个修改原因，代码更清晰、易测试。

### 2、开闭原则 (Open-Closed Principle, OCP)

代码应该对扩展开放，对修改关闭。如果要新增功能，只需要再去实现一个类，而不修改原有代码

**场景**：一个支付系统，初始只支持支付宝，后续需要增加微信支付、银联支付。若使用 `if-else` 判断支付类型，每次新增支付方式都要修改原有类。

```java
// 违反 OCP
public class PaymentProcessor {
    public void pay(String type, double amount) {
        if ("alipay".equals(type)) {
            // 支付宝逻辑
        } else if ("wechat".equals(type)) {
            // 新增时需修改此处
        }
    }
}
```

**符合 OCP 的做法**：定义 `PaymentMethod` 接口，每种支付方式独立实现，通过扩展新增方式，而不修改 `PaymentProcessor`。

```java
public interface PaymentMethod {
    void pay(double amount);
}

public class Alipay implements PaymentMethod {
    public void pay(double amount) { ... }
}

public class WechatPay implements PaymentMethod {
    public void pay(double amount) { ... }
}

public class PaymentProcessor {
    public void process(PaymentMethod method, double amount) {
        method.pay(amount);
    }
}
```

这样新增银联支付时只需新增一个类，对扩展开放，对修改关闭。

### 3、里氏替换原则 (Liskov Substitution Principle, LSP)

子类必须能够替换掉父类，也就是子类必须符合继承关系，而不是为了复用父类的某些代码而错误的继承

**场景**：有一个 `Bird` 类，子类 `Penguin` 重写了 `fly()` 方法抛异常，因为企鹅不会飞。

```java
public class Bird {
    public void fly() { System.out.println("飞"); }
}

public class Penguin extends Bird {
    @Override
    public void fly() {
        throw new UnsupportedOperationException("企鹅不会飞");
    }
}

// 客户端代码
public void letBirdFly(Bird bird) {
    bird.fly();  // 传入 Penguin 会崩溃
}
```

**问题**：子类不能完整替换父类，破坏了程序的正确性。

**改进**：将“会飞”的行为抽象成独立接口 `Flyable`，只有会飞的鸟才实现它。`Penguin` 作为 `Bird` 的子类但不实现 `Flyable`，这样在需要飞行的场景下只传入 `Flyable` 对象，避免非法调用。

### 4、接口隔离原则 (Interface Segregation Principle, ISP)

应该使用多个专门的接口，而不是一个大的接口，客户端不应实现它不需要的接口

**场景**：设计一个 `Worker` 接口，包含 `work()` 和 `eat()` 方法。但某些机器人只需要工作，不需要吃饭。

```java
public interface Worker {
    void work();
    void eat();
}

public class Robot implements Worker {
    public void work() { ... }
    public void eat() { // 空实现或抛出异常 }
}
```

**问题**：机器人被迫实现无意义的方法，接口臃肿，客户端依赖了不需要的方法。

**重构**：将接口拆分为 `Workable` 和 `Eatable`，机器人只实现 `Workable`，人类工作者可以同时实现两个接口。客户端根据实际需要依赖精简接口。

```java
public interface Workable {
    void work();
}
public interface Eatable {
    void eat();
}
public class Robot implements Workable { ... }
public class Human implements Workable, Eatable { ... }
```

### 5、依赖倒置原则 (Dependency Inversion Principle, DIP)

高层模块不应依赖低层模块的实现，而是依赖于接口，从而实现模块的解耦，便于功能的扩展与修改

**场景**：一个 `NotificationService` 直接依赖具体的 `EmailSender` 类。

```java
public class EmailSender {
    public void send(String message) { ... }
}

public class NotificationService {
    private EmailSender emailSender = new EmailSender();  // 依赖具体类
    
    public void notify(String message) {
        emailSender.send(message);
    }
}
```

**问题**：如果需要改用短信或微信通知，必须修改 `NotificationService`，高层模块依赖了低层实现细节。

**重构**：引入抽象 `MessageSender` 接口，高层模块 `NotificationService` 依赖该接口，低层模块实现该接口。并通过依赖注入将具体实现传入。

```java
public interface MessageSender {
    void send(String message);
}

public class EmailSender implements MessageSender {
    public void send(String message) { ... }
}

public class NotificationService {
    private MessageSender sender;
    
    public NotificationService(MessageSender sender) {
        this.sender = sender;
    }
    
    public void notify(String message) {
        sender.send(message);
    }
}
```

现在 `NotificationService` 不关心具体发送方式，扩展性增强，且符合依赖倒置原则（高层与低层都依赖抽象）。
