# 核心的设计模式

## 一、创建型模式

创建型模式关注“对象如何创建”

### 1. 单例模式（Spring）

#### 1.1 作用

单例模式保证在任何情况下，一个类只有一个实例，避免实例被重复创建和销毁，从而节省资源和便于管理

它的构造方法必须是私有的，并对外提供一个全局唯一的访问点

缺点在于违背了单一职责原则，因为它既要负责自身的业务逻辑，又要关心如何对外提供实例化的方法

#### 1.2 饿汉式

若要使用饿汉式，通常是利用类加载机制来保证线程安全（类加载保证只有一个线程执行`static`修饰的代码）

```java
public class Singleton {
    // 1. 私有化构造函数
    private Singleton() {}
    
    // 2. 类加载时就创建实例
    private static final Singleton instance = new Singleton();
    
    // 3. 提供公共访问方法
    public static Singleton getInstance() {
        return instance;
    }
}
```

也可以通过枚举类型，使代码更加简单

```java
public enum Singleton {
    INSTANCE;

    public void doSomething() {
        System.out.println("...doing something");
    }
}

Singleton singleton = Singleton.INSTANCE;
```

饿汉式的缺点在于，在程序刚启动时就创建了所有单例，但它可能不会被使用，导致浪费了空间

#### 1.3 懒汉式

懒汉式通常使用双重检查锁来保证线程安全并提高性能

```java
public class Singleton {
    private Singleton() {}
    // 使用 volatile 保证可见性和禁止指令重排
    private static volatile Singleton instance;

    public static Singleton getInstance() {
        // 第一次检查：避免不必要的同步
        if (instance == null) {
            synchronized (Singleton.class) {
                // 第二次检查：在同步块内部确保只有一个线程创建实例
                if (instance == null) {
                    // new 操作不是原子性的，需要 volatile
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

volatile的作用：

- 保证可见性：也就是保证`instance`字段被初始化之后可以让其他线程立刻看到，防止其他线程第一次检查时看不到变化，又去尝试获取锁（`synchronized`只能保证代码块内的线程可见性，原理是在退出`synchronized`代码块后，清空工作内存）
- 防止指令重排序：`instance = new Singleton();`这行代码可分为，为`instance`分配内存、初始化`instance`、将`instance`指向分配的内存空间三步执行。如果第三步重排序到第二步之前执行，那么其他线程可能提前获取一个未初始化的`instance`

#### 1.4 使用场景

比如Spring Bean的默认作用域

### 2. 工厂方法模式（Spring）

工厂方法模式就是将产品的创建与产品的使用解耦

```java
// 产品接口
public interface Logger {
    void log(String message);
}

// 具体产品：文件日志
public class FileLogger implements Logger {
    @Override
    public void log(String message) {
        System.out.println("FileLogger: " + message);
    }
}

// 具体产品：数据库日志
public class DatabaseLogger implements Logger {
    @Override
    public void log(String message) {
        System.out.println("DatabaseLogger: " + message);
    }
}

// 工厂接口
public interface LoggerFactory {
    Logger createLogger();
}

// 具体工厂：生产文件日志
public class FileLoggerFactory implements LoggerFactory {
    @Override
    public Logger createLogger() {
        // 可以包含复杂的初始化逻辑
        return new FileLogger();
    }
}

// 具体工厂：生产数据库日志
public class DatabaseLoggerFactory implements LoggerFactory {
    @Override
    public Logger createLogger() {
        return new DatabaseLogger();
    }
}

// 客户端代码
public class Client {
    public static void main(String[] args) {
        LoggerFactory factory = new FileLoggerFactory();
        Logger logger = factory.createLogger();
        logger.log("Hello, Factory Method!");

        // 只需修改new这一行，后两行不用改
        factory = new DatabaseLoggerFactory();
        logger = factory.createLogger();
        logger.log("Hello again!");
    }
}
```

比如Spring的IoC容器负责Bean的创建、管理和注入，从而使其依赖者可以直接使用

### 3. 抽象工厂模式

- 工厂方法模式用于解决一个产品的创建问题，比如不同风格的按钮如何创建
- 抽象工厂模式用于解决一组产品的创建问题，比如不同风格的按钮和复选框如何创建，解决的问题更加复杂，并且当需要增加产品等级时，会破坏开闭原则
  - 产品族：一种风格就是一个产品族
  - 产品等级：按钮和复选框就是不同的产品等级

### 4. 建造者模式

建造者模式通过静态内部`Builder`类的方式来快速、灵活的设置对象值，从而避免构造函数参数组合过多的问题，并提高可读性

```java
// 产品：电脑
public class Computer {
    private String cpu;
    private String ram;
    private String storage;
    private String gpu;
    private String motherboard;

    // 私有构造，只允许 Builder 内部调用
    private Computer(Builder builder) {
        this.cpu = builder.cpu;
        this.ram = builder.ram;
        this.storage = builder.storage;
        this.gpu = builder.gpu;
        this.motherboard = builder.motherboard;
    }

    @Override
    public String toString() {
        return "Computer{" +
                "cpu='" + cpu + '\'' +
                ", ram='" + ram + '\'' +
                ", storage='" + storage + '\'' +
                ", gpu='" + gpu + '\'' +
                ", motherboard='" + motherboard + '\'' +
                '}';
    }

    // 静态内部 Builder 类（经典实现，省略 Director）
    public static class Builder {
        // 必需参数
        private String cpu;
        private String ram;
        // 可选参数
        private String storage = "256GB SSD";
        private String gpu = "Integrated";
        private String motherboard = "Standard";

        public Builder(String cpu, String ram) {
            this.cpu = cpu;
            this.ram = ram;
        }
        public Builder storage(String storage) { this.storage = storage; return this; }
        public Builder gpu(String gpu) { this.gpu = gpu; return this; }
        public Builder motherboard(String motherboard) { this.motherboard = motherboard; return this; }
        public Computer build() {
            return new Computer(this);
        }
    }
}

// 客户端使用
public class Client {
    public static void main(String[] args) {
        Computer gamingPC = new Computer.Builder("Intel i9", "32GB")
                .storage("1TB NVMe")
                .gpu("NVIDIA RTX 4090")
                .motherboard("Gaming Z790")
                .build();
        System.out.println(gamingPC);
    }
}
```

### 5. 原型模式

原型模式是通过深拷贝的方式快速复制一个已有对象，从而避免对象初始化时可能要进行的复杂操作，比如读数据库等

原型模式中的原型与Bean的原型作用域的原型不是同一个意思

## 二、结构型模式

关注“类和对象如何组合”

### 1. 适配器模式（JDK）

适配器模式将一个类的接口转换成用户期望的另一个接口，使得原本不兼容的接口能够在原有的类上使用

比如`Arrays.asList()`方法，它把一个数组（Adaptee）适配成`List`接口（Target），使得原本对数组不兼容的`get/set`方法也能够在数组上使用

模拟`Arrays.asList`的核心逻辑：

```java
// 目标接口：List
public interface List<E> {
    int size();
    E get(int index);
    E set(int index, E element);
    // ... 其他方法
}

// 适配者：数组（这里用实际参数传入）

// 适配器类（简化版，模拟 Arrays 的内部类）
class ArrayAdapter<E> implements List<E> {
    private final E[] array;   // 持有适配者（数组引用）

    public ArrayAdapter(E[] array) {
        this.array = array;
    }

    @Override
    public int size() {
        return array.length;
    }

    @Override
    public E get(int index) {
        return array[index];
    }

    @Override
    public E set(int index, E element) {
        E old = array[index];
        array[index] = element;
        return old;
    }

    // 不支持增删
    @Override
    public boolean add(E e) {
        throw new UnsupportedOperationException();
    }

    // ... 其他方法类似
}

// 客户端使用
public class Client {
    public static void main(String[] args) {
        String[] arr = {"A", "B", "C"};          // 适配者
        List<String> list = new ArrayAdapter<>(arr); // 适配器包装数组

        // 现在可以像使用 List 一样使用数组
        System.out.println(list.size());   // 3
        System.out.println(list.get(1));   // B
        list.set(1, "X");
        System.out.println(arr[1]);        // X（原始数组被修改！）
    }
}
```

### 2. 装饰器模式（JDK）

装饰器模式通过组合而非继承的方式，给对象添加新的功能，适合在继承不可用时使用（比如类被`final`修饰）

比如 Java 标准库中的 `java.io` 包大量使用了装饰器模式：

```java
InputStream is = new FileInputStream("test.txt");
// 用缓冲装饰器增强，增加缓冲功能
is = new BufferedInputStream(is);
// 再用数据装饰器增强，增加读取基本数据类型的能力
is = new DataInputStream(is);
```

### 3. 代理模式（Spring）

代理模式就是使用AOP的思想，将与业务逻辑无关的代码抽离出来，达到解耦、复用的目的，比如权限校验、日志收集等

比如Spring中的动态代理

## 三、行为型模式

关注“对象之间如何通信”

### 1. 观察者模式（Spring）

观察者模式使多个观察者关注同一个事件，从而使观察者与被观察者解耦，类似于消息队列的作用

比如Spring中的`ApplicationEvent`与`ApplicationListener`，用于实现事件的发布订阅

### 2. 责任链模式（Spring）

责任链模式是将一个长步骤划分为多个小步骤，每个小步骤作为一条处理链上的节点，同一条处理链上的节点具有相同的标识，还可以通过设置一个order字段决定节点的执行顺序。这种模式能够使代码结构清晰、符合开闭原则、便于复用

比如Spring中的过滤器与拦截器都是责任链模式的应用

### 3. 策略模式（JDK）

策略模式通过组合的方式在运行时动态切换方法的执行策略

比如`Arrays.sort()`方法，可以接收不同的比较器，从而执行不同的排序逻辑

### 4. 模板方法模式（Spring）

模板方法模式是通过继承的方式实现父类中的可变步骤，从而复用父类中定义的算法框架

比如发送MQ消息时，父类中的框架包括组装消息的目的地（topic+tag）、构建消息内容、发送消息、统一的日志与异常处理等。其中对于不同的事件，消息内容的构建方式也不同，就需要具体的子类去实现

比如在Spring中的`JdbcTemplate`，但它是一种基于回调的模版方法模式的变体

策略模式与模板方法模式的选择，主要看是否每次调用都可能使用不同的策略：

- 比如调用sort时可能按升序或降序排序，这种场景如果用模板方法，就要对每种情况实现子类，并用if-else判断再调用，显然不合适。所以这种场景就需要用策略模式，将策略以参数的形式传入，从而灵活的使用不同的策略
- 而MQ发消息的场景中，同一个topic的中的消息构建流程是固定的，所以可以使用模版方法模式
