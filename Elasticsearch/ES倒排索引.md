# ES倒排索引

## 一、倒排索引原理

假设有3条文档数据

| id   | name | age  | sex    |
| ---- | ---- | ---- | ------ |
| 1    | a    | 11   | female |
| 2    | b    | 11   | male   |
| 3    | c    | 12   | male   |

那么Elasticsearch建立的索引如下

name：

| term | posting list |
| ---- | ------------ |
| a    | [1]          |
| b    | [2]          |
| c    | [3]          |

age：

| term | posting list |
| ---- | ------------ |
| 11   | [1, 2]       |
| 12   | [3]          |

sex：

| term   | posting list |
| ------ | ------------ |
| female | [1]          |
| male   | [2, 3]       |

Elasticsearch分别为每个field都建立了一个倒排索引，a、b、c、11、12、female、male这些叫做词项term，而[1,2]就是倒排列表Posting List，倒排列表中存储了出现该词项的文档的id

词项的集合叫做词典，**词典通过B+树存储**，原理类似MySQL索引

**总结**：倒排索引就是先通过分词得到不同的词项，然后建立词项与文档id的映射关系。在查找某一词项时，可直接获得所有包含该词项的文档id，从而提高查找速度

## 二、优点

- 适合多条件查询：在多条件查询时，ES对每个条件进行一次倒排索引查询，得到多个文档id的集合后做交集运算，从而得到符合所有条件的文档（避免了MySQL需要对每种查询条件都建立索引，MySQL只适合条件简单的查询）

## 三、缺点

- 一致性弱，不保证事务，不适合承担写操作
- 索引的存储成本比MySQL更大，是一种空间换时间的策略
