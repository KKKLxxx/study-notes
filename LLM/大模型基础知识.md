## Transformer原理

**一句话介绍**

Transformer是一种基于注意力机制的序列建模架构，它的核心在于：

- 第一，它通过QKV机制直接建模任意两个 token 之间的关系
- 第二，它通过多头注意力从不同角度去理解同一句话，从而增强表达能力
- 第三，它通过位置编码弥补Transformer对顺序不敏感的特点

它相比传统神经网络的最大优势在于，能够直接建模远距离的依赖关系，而无需通过增加神经网络的层数逐步扩大感受野，从而保留了信息的精度

**什么是序列建模架构？**

指的是专门用于处理“有顺序关系的数据”的模型，比如文本、语音、时间序列等。模型的目标是建模元素之间的依赖关系，例如“当前词和前后词之间的关系”

**注意力的本质是什么？**

是一种“加权的信息选择机制”，它让模型根据当前查询Q与各个部分K的匹配程度计算出权重，然后与信息内容V相乘，得到最相关的信息

- Query（我想找什么）
- Key（我能提供什么信息）
- Value（真正的信息内容）

注意力计算公式：$\mathrm{Attention}(Q,K,V)=\mathrm{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V$

## Agent设计范式

设计范式指的是大模型如何做决策与执行

- ReAct 是一种让模型在推理和行动之间交替进行的模式，它通过 Thought、Action、Observation（思考、行动、观察） 的循环，实现边推理边调用工具，适合需要动态决策的场景
- Plan-and-Execute 是先生成完整计划，再按步骤执行的模式，更适合复杂、多步骤任务

两者的核心区别在于：ReAct 是“边想边做”，强调灵活性；Plan-and-Execute 是“先想后做”，强调可控性

在实际应用中，通常会把两者结合，比如用 Plan 做任务拆解，用 ReAct 执行具体步骤

## LangChain与LangGraph

LangChain主要封装了模型调用相关的接口，能够以有向无环图的线性工作流完成任务

LangGraph建立在LangChain的基础之上，可以通过节点路由来实现ReAct模式的循环工作流，从而完成更加复杂的任务

## OpenClaw

openclaw是一个agent的网关，它能够将任务交给某个agent去处理。agent会调用skill完成任务，执行skill时会通过mcp发现任务执行时所需要的工具，然后agent通过function calling执行工具调用

## 多Agent之间的协作怎么实现的

多Agent协作主要需要解决任务的拆解与分配、通信以及冲突处理3方面问题：

- 任务拆解与分配：让一个主Agent进行进行任务的拆解和分配，或者拆解后让其他Agent根据自身能力进行任务的竞标（存在去中心化的方式，但实际上是弱中心化，也就是每个Agent都具备拆解和分工能力，自己只处理能处理的部分，将剩余任务交给其他Agent）
- 通信：可以通过共享内存或消息传递方式进行通信
- 冲突处理：可能发生资源抢占等问题，可以通过锁机制或优先级策略来解决

## Vibe Coding流程，如何保证代码质量

对于一个任务，我一般会分成4步去完成，分别是确认需求、确认方案、编码实现、结果验证

- 确认需求时，我会要求它主动询问不明确的点，而不是自行假设，这样能够减少歧义，从源头控制错误
- 确认方案时，我会要求它给出多个方案并说明trade-off
- 编码实现时，我会要求它同时编写单元测试
- 结果验证时，我会要求它根据测试结果进行自动修复，并在最后让AI进行一次自检，寻找潜在bug和边界情况

我更习惯于用cursor开发，因为cursor能够实时看到生成的代码，这样对代码的控制会更强

## Prompt Engineering

提示词工程是让AI明确任务目标

一个好的提示词至少要包含3部分内容，分别是：

- 角色与任务：角色与任务中除了要向AI说明你是谁、你要完成什么任务，还要向AI补充必要的业务背景
- 约束条件：约束条件通常包含回答的风格、字数限制、禁止回答的内容等
- 输出格式：输出格式可以要求以JSON格式输出，或者要求输出内容必须添加引用等，可以通过few-shot（少样本示例）来让模型更明确地理解意图

最后还要对提示词进行迭代，需要从bad case中分析原因，并构建完整的测试集和测试流程评估优化效果

## Harness Engineering
