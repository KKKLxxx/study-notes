# 缓存穿透、缓存击穿、缓存雪崩

## 一、缓存穿透

### 1. 发生场景

高并发访问并不存在的key，由于不可能缓存命中，导致每次都会访问数据库，从而导致数据库崩溃

### 2. 解决方案

#### 2.1 布隆过滤器

- 布隆过滤器维护了一个二进制数组（HashMap空间占用太大）

  - 插入时，通过多个hash函数对key计算得到多个哈希值，并对数组长度取模，得到应置1的位置
    - 哈希函数的个数与数组长度、预期的插入量和错误概率相关，个数太少或太多都会导致误判率增加（太少会导致数据分布不均匀，太多会导致数组中1的个数增加），一般是10个左右
  - 查找时，如果对应下标的值均为1，则认为数据存在，反之认为不存在

- 布隆过滤器的优点在于

  - 占用空间小
  - 判定速度快

- 布隆过滤器的缺点在于

  - 有一定的误判率，只会将不存在的误判为存在，而不会反过来
    - 因此如果只通过布隆过滤器防止缓存穿透，极端情况下仍有可能根据接口的响应速度来判断是否被布隆过滤器有效过滤，从而以没有被有效过滤的数据作为参数继续访问，再次造成缓存穿透问题（具体来讲，如果返回数据不存在的速度很快，说明是被布隆过滤器检测出来了；如果速度较慢，说明没有被过滤，而是仍然访问了数据库，才检查出来数据不存在）因此还要引入空值缓存进一步解决

  - 不能删除元素
    - 如果数据存在过期时间，大量无效的数据会存在于布隆过滤器当中，导致误判率上升。可以通过一个后台线程定期重建布隆过滤器

- 可通过Redisson快速创建布隆过滤器，需要指定预期的插入量和错误概率

  ```java
  @Configuration
  public class RBloomFilterConfiguration {
      /**
       * 优惠券查询缓存穿透布隆过滤器
       */
      @Bean
      public RBloomFilter<String> couponTemplateQueryBloomFilter(RedissonClient redissonClient) {
          RBloomFilter<String> bloomFilter = redissonClient.getBloomFilter("couponTemplateQueryBloomFilter");
          bloomFilter.tryInit(640L, 0.001);
          return bloomFilter;
      }
  }
  ```

- 布隆过滤器的容量有上限，如果一个布隆过滤器无法存储所有数据怎么办？
  - 根据数据的有效期去重建布隆过滤器
  - 如果有效期内的数据也在上限之上，可以对布隆过滤器进行分片，创建多个过滤器。也就是先根据数据的哈希值计算出应在哪个过滤器查询，再在对应的过滤器中查询

#### 2.2 空值缓存

空值缓存用于解决因布隆过滤器可能误判的问题而导致的漏洞

具体做法是，当从数据库中查找到数据不存在时，会设置一个有效时间较短的key，表示对应的数据不存在。如果其他请求同时查找这个不存在的数据，就会被空值缓存直接过滤，从而不再请求数据库

#### 2.3 分布式锁+双重检查锁

分布式锁是为了在多线程环境中，只有一个线程能够去执行查找数据库的操作

双重检查锁是为了避免后面的线程重复地执行查数据库的操作，而是可以从已更新的缓存中获取数据（要检查缓存key与空值key）

**极端高并发场景**：

即使加入了双重检查锁，也有可能同时有很多线程第一次检查全部为空，然后去参与锁竞争，这就会导致大量线程阻塞在获取锁的操作上。解决方案：

- 通过tryLock()尝试获取锁：如果获取失败，则做一个降级处理，返回“当前访问人数较多，请稍后重试”等内容，但这种处理方式用户体验会比较差
- 分布式锁分片：由于只有一把锁，所以数据库只允许一个线程访问，但其实数据库这种资源是允许同时读的，所以可以根据线程ID对锁进行分片。假设允许10个线程同时获取不同的分布式锁，就能将锁竞争降低10倍

但这些方案一般只存在于理论上

## 二、缓存击穿

### 1. 发生场景

高并发访问一个存在的热点key，但此时缓存失效，仍会每次都访问数据库

所以缓存击穿要解决的问题就是在缓存失效到恢复缓存这段时间，将请求阻塞，防止所有请求都访问数据库

### 2. 解决方案

#### 2.1 热点key不要设置过期时间

虽然设置热点key永不过期能够有效解决缓存击穿问题，但是由于内存淘汰策略以及数据更新问题，也可能造成热点key失效的情况，所以要进一步保障代码的健壮性

其他解决方案（这里没考虑数据更新问题）：

- 不给key设置过期时间，淘汰策略选择volatile类型，再通过MQ延迟消息保证key不会再被使用后再删除。但这会极大增加系统复杂度
- 设置过期时间，淘汰策略选择volatile-lfu（比lru更准确），结合其他策略避免缓存击穿

#### 2.2 缓存预热

缓存预热就是在将数据插入到数据库的时候，直接将其更新到缓存当中，为后面可能的查询请求做准备

#### 2.3 分布式锁+双重检查锁

同上

## 三、缓存雪崩

### 1. 发生场景

缓存服务宕机或大量缓存同时失效，导致数据库访问量过大

### 2. 解决方案

应对缓存服务宕机问题：

- 通过建立集群确保缓存服务高可用

应对大量缓存同时失效问题：

- 通过随机数将过期时间尽量分散
