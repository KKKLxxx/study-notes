# Redis分布式锁

## 一、作用

在一个分布式系统中，多台机器上部署了多个服务，当客户端一个用户发起一个数据插入请求时，如果没有分布式锁机制保证，那么那多台机器上的多个服务可能进行并发插入操作，导致数据重复插入。而分布式锁机制就是为了解决类似这类问题，**保证在分布式环境下，多个服务可以互斥地访问共享资源**，如果一个服务抢占了分布式锁，其他服务没获取到锁，就不进行后续操作

## 二、实现方式

实现分布式锁的核心步骤为六步：

### 1. 通过`SET NX PX`加锁

`NX`保证只有`key`不存在的时候才能添加

`PX`通过设置过期时间保证即使服务宕机锁也可以正常释放

### 2. 保证只有锁的拥有者可以释放锁

可以通过将`value`设置为线程或者服务的唯一ID，并在释放锁之前检查`value`来实现

### 3. 保证锁可重入

也就是需要记录锁的获取次数，可以将`key`设置为Hash类型来同时保存拥有者和获取次数两个属性

### 4. 通过看门狗（Watch Dog）机制延长锁的过期时间

因为默认的过期时间可能不够长时间任务的执行，所以要通过一个后台线程检查任务是否执行完毕，如果没有，则要延长过期时间

### 5. 保证锁释放的原子性

也就是需要通过`Lua`脚本，将检查`value`和删除`key`这两个操作原子地执行

### 6. 保证集群模式下的数据一致性

在集群模式下，如果主节点执行完`SET`指令后，还没来得及让从节点同步就宕机了，那么集群会重新选举一个主节点，那么就有可能存在多个服务同时拥有锁的情况

RedLock提供的解决方案是，让客户端与多个主节点建立连接，只有从多数主节点中成功获取锁，且总耗时小于锁过期时间，才算真正获取锁

但这种方案性能差、实现复杂，实际中一般不会使用（除非金融场景要求强一致性），而是直接使用Redisson的单key方案

## 三、分布式锁的粒度控制

分布式锁的粒度控制是通过key的命名来实现的，比如：

- 如果要实现接口的全局限流：key中只用包含接口名称
- 如果要实现接口对某个用户限流：key要包含接口名称和用户ID
- 如果要实现接口的防止重复提交：key要包含接口名称、用户ID、提交的内容（内容相同才会被认为是重复提交）
