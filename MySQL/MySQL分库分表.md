# MySQL分库分表

## 一、垂直/水平扩展

### 1. 垂直分表

将一张列很多的宽表，按列的访问频率拆成多张表，通过主键关联

主要作用：提升热点列的查询效率

### 2. 垂直分库

按业务模块将不同的表拆分到不同的数据库，让不同模块连接不同的数据库

### 3. 水平分库分表

按分片键的哈希或范围，将数据分散到多个库的多个表中

## 二、分库与分表的区别

分库解决的是数据库所在服务器的连接数、CPU、磁盘容量、磁盘IO等方面的问题

分表解决的是单表数据过多导致的索引深度加深、锁竞争等问题

## 三、单表行数限制

### 1. 单表数据量上限

MySQL是没有对单表数据量进行限制的，其上限取决于系统对文件大小的限制和硬盘容量的限制

《阿里巴巴Java开发手册》提出单表行数超过 500 万行（3次磁盘IO）或者单表容量超过 2GB，才推荐进行分库分表

更准确的应该是通过实际的查询耗时来判断是否需要分表

### 2. 数据量过大时为什么查询会变慢

1、数据量增多时，会使B+树的高度增加，磁盘IO次数也就更多

2、数据量过大，会导致锁竞争更激烈

3、可能导致Buffer Pool缓存命中率下降，也就需要重新从磁盘中读取

#### 2.1 Buffer Pool

由 InnoDB 存储引擎提供

作用：将磁盘上的数据页和索引页缓存在内存中。读取数据时优先从内存取，写入数据时先写内存中的脏页，再由后台线程异步刷新到磁盘（WAL 机制）

查询流程：

```
SQL → 先查 Buffer Pool（内存）

      命中 → 直接返回，不读磁盘

      未命中 → 从磁盘加载到 Buffer Pool → 返回
```

Buffer Pool 内部使用改进的 LRU 算法管理页的淘汰，分为 Young 区（热数据）和 Old 区（冷数据），防止全表扫描把热数据全部挤出

### 3. 自增id超出上限的问题

假设以一个int类型的id作为自增主键，int的上限是2^31-1(即2147483647)，那么当存在id=2147483647的数据后，再插入，会显示`Duplicate entry`，即出现重复的键，然后禁止插入

所以可以认为，当自增id到达上限后，数据的id不会增加，会重复使用最大的id进行插入。但因为主键已经存在，所以会插入失败
